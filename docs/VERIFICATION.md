# Проверка отображения

Как убедиться, что часы всё ещё выглядят правильно после правки. Проверка нужна потому, что три вещи в этом проекте невозможно надёжно оценить на глаз, и все три уже ломались.

## Что проверяется и почему именно это

**Часы влезают в экран и заполняют его.** Размер считает JavaScript, а не CSS, поэтому ошибка в пропорции не подсвечивается ни в каком редакторе: она просто выезжает за край на одной модели iPad и оставляет полэкрана пустым на другой. Проверяются все пять индикаторов на пяти разрешениях в трёх конфигурациях - 75 раскладок.

**Дата светится так же, как цифры.** Цифры нарисованы коробками, дата - текстом, поэтому ничто не держит их на одной яркости автоматически. Расхождение уже случалось дважды: сначала текст был ярче времени, потом темнее. Скрипт замеряет пиковую светимость ядра штриха и сравнивает.

**Как это выглядит.** То, что число не расскажет: обрезался ли ореол, не превратилась ли деталь в грязное пятно на большом размере, не рассыпалась ли колба лампы.

## Запуск

Скрипту нужен Node, самим часам - нет. Ничего из `tools/` на iPad не попадает.

```bash
cd tools
npm install
node check.js
```

Отдельные проверки:

```bash
node check.js fit         # раскладки: переполнение и пустой экран
node check.js brightness  # яркость даты против цифр
node check.js shots       # скриншоты каждого режима в tools/shots
```

`node check.js` возвращает ненулевой код, если что-то не сошлось, - его можно повесить в CI.

## Как читать результат

```text
FIT: all 75 layouts fit and fill the screen
```

Раскладка помечается плохой, если часы вылезают за экран (жёсткая ошибка) или занимают меньше 90% его ширины (мягкая: значит, где-то потеряли размер).

```text
ok       vfd/ice: digit 240, date 238, ratio 0.99
MISMATCH led/red: digit 188, date 138, ratio 0.73
```

`ratio` - отношение яркости даты к яркости цифры. Допуск 0.85-1.15: за его пределами два элемента читаются как куски разных дисплеев. Замер берёт не средний пиксель области (его утопили бы тёмные промежутки между штрихами) и не самый яркий (это был бы артефакт сглаживания), а среднее по 2% самых ярких пикселей - то есть тело светящейся фигуры.

Часы в скрипте остановлены на 10:08:42. Это не украшение: время должно быть вне окна ночного затемнения, иначе `opacity: 0.5` тихо вдвое занизит все замеры яркости.

## Ручная проверка на самом iPad

Численно всё это проверяется в headless Chrome, но старый Safari - другой рендерер. После заметной правки CSS стоит открыть часы на самом устройстве и посмотреть:

- не пропали ли скошенные концы сегментов (`clip-path` с префиксом `-webkit-`);
- не пропало ли свечение (`box-shadow` на `:before`);
- не исчезли ли цифры в лампах (`-webkit-text-stroke`);
- не тормозит ли перерисовка секунд.

Если изменения не видны вообще - это почти всегда офлайн-кэш, а не CSS: см. раздел про версии в [DEVELOPMENT.md](DEVELOPMENT.md).
